Inside the AWS Architecture: How this Cloud Resume Project Works

This project demonstrates a minimal, production-ready serverless static website deployed on AWS with a small dynamic component (a visitor counter). It prioritizes low operational overhead, secure defaults, and repeatable infrastructure through Terraform.

Static Hosting: S3 + CloudFront

All static assets (HTML, CSS, JS, images) are stored in an Amazon S3 bucket configured for website hosting. CloudFront sits in front of S3 as the CDN, providing TLS termination, global edge caching, and improved performance for visitors worldwide. CloudFront also allows cache-control headers and path-based behaviors, so we can serve static content fast while forwarding specific API calls to the dynamic backend.

Visitor Counter: Lambda + API Gateway + DynamoDB

The only dynamic piece is a lightweight Lambda function that atomically increments and returns the visitor count. The function is invoked via a public API (API Gateway). It uses DynamoDB as a durable key-value store with a single-item counter record. The Lambda code performs an atomic update (UpdateItem with ADD) to avoid race conditions under concurrent access.

Infrastructure as Code

Terraform defines S3, CloudFront, Lambda, DynamoDB, IAM roles, and API Gateway configurations. This ensures resources are reproducible, auditable, and easy to change. IAM policies follow the least-privilege principle: the Lambda role has permission only to read/write the specific DynamoDB table and write logs to CloudWatch.

CI/CD — GitHub Actions

A GitHub Actions workflow builds and deploys the site on push to the main branch. The job syncs compiled site files to S3 and then triggers CloudFront invalidation so edge caches update quickly. Terraform runs can be included in the workflow (plan/apply) behind appropriate safeguards (branch protections and PR approvals).

Security & Performance Considerations

  • HTTPS everywhere via CloudFront — no plaintext HTTP exposed to users.
  • Cache static assets aggressively with sensible cache-control headers to reduce origin egress and improve latency.
  • Use DynamoDB provisioned capacity or on-demand depending on expected traffic; for small projects, on-demand reduces management overhead.
  • Enable CloudWatch logging and set basic alerts for Lambda errors or high 5xx rates from API Gateway.

How a Visit Works (Step-by-step)

  1. User requests the homepage — CloudFront serves cached content from the nearest edge. If stale, CloudFront fetches from S3.
  2. The page's JavaScript calls the counter API endpoint (POST to /count).
  3. API Gateway receives the request and forwards it to Lambda.
  4. Lambda updates the visitor count in DynamoDB using an atomic update and returns the new value.
  5. The page displays the returned count — visible to the user immediately.

Observability & Cost

CloudWatch collects logs and metrics from Lambda and API Gateway. For small personal projects, costs are minimal — S3 and CloudFront for storage and delivery, DynamoDB on-demand for a single-counter item, and Lambda requests billed at a small fraction of a dollar per month for low traffic.

Extensibility

The architecture is intentionally modular: you can add authentication, richer APIs, analytics, or serverless functions without changing the static hosting pattern. Terraform modules make it easy to add environments (staging/production) and reuse components.
